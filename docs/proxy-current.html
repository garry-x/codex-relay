codex-relay 0.4.0 / 当前 proxy 方案

VPS Forwarder + Tailscale + Static Proxy

当前方案把本机 Codex CLI 的代理请求送到 Tailscale 内网里的 VPS，再由 VPS 上的轻量 CONNECT forwarder 通过上游静态代理访问 OpenAI。全程不做 TLS 终止，VPS 只转发隧道字节流。

一、链路总览

本机
  codex-relay run
    |
    | 注入 HTTP_PROXY / HTTPS_PROXY
    v
  Codex CLI
    |
    | HTTP CONNECT api.openai.com:443
    | 目标代理: http://100.114.41.104:8443
    v
Tailscale / WireGuard 内网
    |
    v
VPS
  codex-relay-forward 0.0.0.0:8443
    |
    | 校验 CONNECT 目标是否在 FORWARD_ALLOW
    | 重新向上游静态代理发起 CONNECT
    v
上游 Static Proxy
    |
    | TLS 隧道继续透传
    v
OpenAI / Codex 相关服务

请求时序

  1. 用户执行 codex-relay run ... 或直接通过 codex-relay <args> 透传。
  2. codex-relay 从环境变量或 ~/.codex-relay/config.json 解析代理地址，并注入 HTTP_PROXY、HTTPS_PROXY、WS_PROXY、WSS_PROXY。
  3. Codex CLI 对 api.openai.com:443 等目标发起 HTTP CONNECT，请求先进入 VPS 的 Tailscale IP 100.114.41.104:8443。
  4. codex-relay-forward 解析 CONNECT 目标，确认命中允许列表后，连接上游静态 HTTP 代理。
  5. VPS forwarder 向上游代理发送新的 CONNECT 请求。上游返回 200 Connection Established 后，本机和 OpenAI 之间的 TLS 数据开始双向 pipe。
  6. VPS forwarder 不读取业务 payload，不解密 HTTPS，不修改 OpenAI 请求体和响应体。

二、组件职责

codex-relay

本机 CLI 包装器。负责保存代理配置、检查连通性、安装 Codex CLI、启动 Codex 时注入代理环境变量。

codex-relay-forward

VPS 上的 TCP CONNECT 转发器。只支持 HTTP 上游代理，默认监听 127.0.0.1:8443，部署时通常显式监听 Tailscale IP 或 0.0.0.0:8443。

Tailscale

本机到 VPS 的 WireGuard 加密网络。避免把 forwarder 暴露到公网，也不需要 SSH 隧道常驻。

关键配置

本机 ~/.codex-relay/config.json
{
  "http": "http://100.114.41.104:8443",
  "https": "http://100.114.41.104:8443"
}

VPS ~/.codex-relay/config.json
{
  "http": "http://user:pass@static-proxy:8080",
  "https": "http://user:pass@static-proxy:8080"
}
环境变量作用
HTTP_PROXY / HTTPS_PROXY本机优先级最高的代理来源。存在时会覆盖配置文件。
FORWARD_PROXYVPS forwarder 的上游静态代理地址，优先于 VPS 配置文件。
FORWARD_LISTENVPS forwarder 监听地址，默认 127.0.0.1:8443。
FORWARD_ALLOW允许 CONNECT 的目标列表，默认包含 api.openai.com:443、chatgpt.com:443、auth.openai.com:443、cdn.openai.com:443。

四、优势

  • 不需要本地 CA 和 TLS 终止。 HTTPS 仍是 Codex CLI 到 OpenAI 的端到端 TLS，VPS 不解密业务流量。
  • 运行面很小。 两个脚本都只依赖 Node.js 内置模块，没有服务端 npm 包和额外守护框架。
  • 部署路径清晰。 本机只知道 VPS Tailscale IP，VPS 只知道上游静态代理，职责分层明确。
  • 比 SSH 隧道更稳定。 依赖 Tailscale 常驻网络，不需要为每次使用维护 SSH tunnel。
  • 有基础目标白名单。 forwarder 默认只允许 OpenAI / Codex 相关域名，降低被误用为通用代理的风险。
  • 诊断直接。 codex-relay proxy check 同时检查代理配置、curl、代理连通性、npm、Codex CLI 和 VPS forwarder 状态。

五、当前问题和风险

  • 链路比直连多两跳。 请求路径包含 Tailscale VPS 和上游静态代理，延时受 VPS 地域、Tailscale DERP/直连状态、上游代理质量共同影响。
  • 上游只支持 HTTP proxy。 codex-relay-forward 的 parseProxy 明确拒绝非 http: 上游，暂不支持 SOCKS、HTTPS proxy。
  • VPS 监听面需要控制。 当前状态显示 forwarder 监听 0.0.0.0:8443。如果 VPS 公网安全组也放开该端口，建议改为只监听 Tailscale IP，或用防火墙只允许 tailnet 来源。
  • allowlist 需要随 Codex 访问域名更新。 新增 OpenAI/Codex 域名时，可能需要调整 FORWARD_ALLOW，否则 forwarder 会返回 403 Forbidden。
  • 空闲超时为 30 秒。 forwarder 对客户端和上游连接设置 30000ms idle timeout。长时间无流量的连接可能被断开。
  • 现有 HTML 指南存在旧模式描述。 docs/guide.html 中还有 chain relay/token 方案内容，但当前 codex-relay 代码没有对应命令；以本文档和 README.md 的 VPS forwarder 方案为准。
